Grocery Item Engine

Once the controller and accessor had been created for the grocery item model I was tasked with creating the engine. The engine is used in MVC architechure to seperate the logic performed from the ui. The engine uses the methods found in the accessor to create seperation between the controller and the accessor. The grocery item controller calls the grocery item engine to use the methods that are located in grocery item accessor. This also provides an example of the Open Closed Principle where the controller uses the engine as an extension of the accessor so nothing can be changed or modified. You can see many of the methods in the engine are similiar to the accessor functions, but a little more descriptive. This helps define the methods better so its easier to tell which method youre using in the controllers when implementing the engine. I also was in charge of creating the interfaces for the engines (more on that later), but at the top of the engine you can see I also implemented the interface to provide dependency. You can download a copy of the Item engine here.

Interfaces

Because the interfaces are relatively small I am going to include both the list and item engine interfaces, and the list and item accessor interfaces as one code piece. After creating the Engines I went ahead and created both the item and list interfaces to go with their respective engines to provide more abstraction and seperation between the front end and back end while also providing more modularity. This allows the devlopers to change something in the project without having it conflict with other parts of the project.

In the examples you can see that the interfaces just include the declaration of the methods and their parameters from the engines. Interfaces are also used in the SOLID Principlesa for the Interface Segragation Principle. This principle says that many interfaces should be created instead one "fat" interfaces, so clients aren't forced to implement interfaces that have methods they dont need or use. In this case I created an interface for each model and each engine/accessor. On the right is actually an example of the Grocery item accessor interface which helps provide abstraction and seperetion from the accessor when its getting called by the engines. I felt that overall the interfaces were easy to make and only deserved to be listed as 1 code piece. Below you can find the several interfaces I created.

  • Grocery List Engine Interface
  • Grocery Item Accessor Interface
  • Grocery List Accessor Interface

Custom Error / 404 Page

This is 1 of the only front end peices of code I worked on and it is just a simple custom 404 page that allows lost users to better understand what happened and use links loacted on the page to navigate back to the intended features. If you attempt to go to a page that either doesn't exist or is missing then you will end up on the custom 404 page. I would say this is related to a few topics that we had talked about during class. The first one being security since you want to cover all the edge cases to prevent poeple from breaking your software by doing something the devolper had not thought of. The second topic from class it touches is UI design because the custom 404 page will easily allow lost users to navigate back to the inteded features using the links on the page. Finally, the third topic that I think it is somewhat related to is unit testing because this allows us to write unit tests and determine if the controllers correctly handle an error. You can download and view the code for the custom error page here.
